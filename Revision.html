Définissons les termes

BDD

  • C’est quoi un ORM ? Object Relational Mapping ==> couche logicielle qui fait l’interface entre le monde objet et le monde relationnel.
  • C’est quoi une clé primaire ? ==> une donnée qui permet d’identifier de manière unique une entrée d’une table.
  • C’est quoi une clé étrangère ? ==> une donnée qui permet d’identifier une relation avec une entrée d’une autre table.
  • C’est quoi une contrainte d’intégrité ? ==> ce qui permet l’intégrité des données => ex : un champ non null, un champ unique, une relation avec une autre table

Entraînement aux requêtes SQL :

Url de connexion : jdbc:h2:mem:star-wars

  • Récupérer toutes les planètes
SELECT * FROM planets;
  • Récupérer seulement les planètes ayant plus de 1.000.000 de population
SELECT * FROM planets
WHERE population > 1000000;
  • Récupérer seulement les planètes ayant plus de 1.000.000 de population triées par population décroissante puis par ordre alphabétique croissant (sur le nom)
SELECT * FROM planets
WHERE population > 1000000
ORDER BY population desc, planet_name asc;
  • Compter le nombre de terrains par planète
SELECT planet_name, COUNT(planets.id) AS nb FROM planets 
JOIN planet_terrains ON planets.id = planet_terrains.planet_idx
JOIN terrains ON planet_terrains.terrain_idx = terrains.id
GROUP BY planets.id;
  • Récupérer seulement les planète ayant plus de 3 terrains différents
SELECT planet_name, COUNT(planets.id) AS nb FROM planets 
JOIN planet_terrains ON planets.id = planet_terrains.planet_idx
JOIN terrains ON planet_terrains.terrain_idx = terrains.id
GROUP BY planets.id HAVING nb > 3;

Web

  • Si on développe une SPA, où s’exécute le code Angular ? ==> Dans le navigateur ! Le serveur Node.js que l’on utilise en mode développement est utilisé seulement pour servir les fichiers HTML, CSS et JS que Angular a compilé pour nous. Ensuite le code JS dialogue directement avec notre serveur back par l’intermédiaire du navigateur.
  • Si on développe une API avec Spring boot, où s’exécute le code Spring Boot ? ==> Sur notre serveur d’application. Dans le cas de Spring Boot, c’est un serveur Tomcat qui est embarqué par Spring Boot.

Ensuite :

  • C’est quoi une API REST ? Application Programming Interface : c’est un ensemble de méthodes qui sont mises à disposition et que l’on peut réutiliser dans d’autres programmes.
    REST : Representational State Transfer ==> architecture à respecter
    • Ressources : URI qui identifie nos ressources (avec nom de ressource au pluriel) ==> ex : /api/planets
    • Verbes : GET pour lire, PUT pour mise à jour, POST pour créer et DELETE pour supprimer.
  • De quoi est composée une requête HTTP ? ==> La ligne de requête (méthode, URL, version HTTP), les en-têtes (ex : Authorization), le corps de la requête (les données à envoyer au serveur).
  • De quoi est composée une réponse HTTP ? ==> Le code de retour (est-ce que ça s’est bien passé ou pas), Les en-têtes de la réponse, le corps de la réponse (les données que nous retournent le serveur).
  • Si on développe en php, comment fonctionne l’application (où sont générées les pages, où s’exécute le php) ? Les pages sont générées côté back en fonction des données reçues par le serveur et seront envoyées au client pour affichage.

Sécurité

  • Authentification vs Authorisation ? ==> “Authentification”: Mécanisme/système sécurisé par lequel on identifie l’utilisateur. “Authorisation”: Mécanisme/système qui détermine le niveau d’accès aux ressources d’un utilisateur authentifié
  • C’est quoi un hash ? ==> Une empreinte numérique d’une donnée. Elle est créée grâce à des fonctions mathématiques bien particulières. Ces fonctions doivent assurer : qu’il est “presque” impossible de retrouver la donnée initiale à partir de l’empreinte, qu’il doit être “presque” impossible de trouver deux messages aléatoires donnant la même empreinte.
  • Comment fonctionne JWT (pour ceux qui l’utilisent) ? ==> JWT est une “carte d’identité numérique” qui permet d’authentifier un utilisateur. Lorsqu’une SPA veut permettre à un utlisateur de s’authentifier, elle appelle un serveur d’authorisation qui va créer le token si l’utilisateur fournit les bons identifiants. Ensuite pour chaque requête vers le serveur ressource, la SPA envoie le token pour que le serveur puisse reconnaître l’utilisateur. Dans le cas du projet cinéma, le serveur d’autorisation et d’authentification sont les mêmes
  • C’est quoi la faille XSS ? ==> C’est une injection de code (JS, Java, …) dans une page web.
  • C’est quoi une injection SQL ? ==> C’est le fait de récupérer des ordres SQL non souhaités dans une entrée utilisateur (ex : un champ de recherche).
  • C’est quoi la faille CSRF ? ==> Article Wikipédia. Pourquoi on peut désactiver la protection dans le cas de l’utilisation de JWT ? ==> Parce qu’on stocke le token dans le session storage.
  • C’est quoi le CORS et pourquoi on en a besoin en développant une SPA ? ==> C’est le fait de refuser ou accepter les requêtes venant d’une autre origine. Dans le cas du développement d’une SPA Angular, notre code est servi au navigateur par un serveur Node sur le port 4200 et notre API écoute sur le port 8080 du localhost. Par principe de sécurité, les requêtes partant du code front vont être bloquées par défaut car elle ne ciblent pas le même origine. Il faut donc autoriser le partage de ressource (Resource Sharing) entre différentes origines (Cross Origin)
